# ClusterPilot 项目说明书

## 1. 项目定位

**项目名称**：ClusterPilot  
**项目类型**：面向 AI 工作负载的 GPU 感知任务调度与资源控制项目  
**目标岗位**：AI Infra、云原生平台、集群调度、分布式系统、GPU 资源管理、MLOps 平台

ClusterPilot 不是 Kubernetes 的替代品，也不是一个“管理 GPU 的后台系统”。它的目标是完成一个面向 AI 训练、在线推理和离线批处理任务的调度控制面原型，并通过模拟或小规模环境证明：

- 不同调度策略如何影响 GPU 利用率、排队时间、公平性和任务完成率；
- 多 GPU 任务为何需要整体调度；
- 高优先级任务到来时，资源如何被回收或让渡；
- 多租户环境下，如何避免少数用户长期垄断稀缺 GPU；
- 为什么 AI 集群的调度不能只按 CPU、内存和单个容器的资源请求来处理。

它应与 BlockServe 形成连续叙事：

> BlockServe 研究单卡内的推理运行时；ClusterPilot 研究多任务、多节点条件下的 GPU 资源控制和调度策略。

---

## 2. 要解决的问题

AI 工作负载与普通 Web 服务不同，主要体现在：

- 任务常常需要多个 GPU 同时启动；
- GPU 之间的拓扑关系会影响通信效率；
- 训练任务周期长、可 checkpoint，推理任务对延迟敏感；
- 不同用户、团队和优先级会竞争有限 GPU；
- 集群很容易出现“总资源还有，但找不到连续或合适资源”的碎片化问题；
- 某些任务可以被抢占和恢复，某些任务则不宜中断。

ClusterPilot 要回答：

- 一个需要 8 张 GPU 的任务何时应该启动？
- 资源不足时，系统是等待、拆分、拒绝还是抢占低优先级任务？
- 多个用户竞争时，如何在效率与公平之间平衡？
- 是否应该优先把同一任务放到同一节点或同一高带宽域？
- 如何把调度决策解释给用户和运维者？
- 如何用数据证明某种策略优于简单 FIFO？

---

## 3. 项目目标与非目标

### 3.1 核心目标

- 建立可描述 GPU 节点、设备能力、拓扑、任务需求与租户约束的资源模型；
- 支持 AI 训练、在线推理、离线批处理三类任务；
- 支持多用户或多队列的资源配额和优先级；
- 支持基本的 GPU 资源放置策略；
- 支持多 GPU 任务的整体调度；
- 支持低优先级任务让出资源的概念验证；
- 支持任务失败、节点不可用和重试等控制面行为；
- 提供模拟器、工作负载回放和策略对比报告；
- 在后期可选择性接入 Kubernetes 作为执行环境或适配目标。

### 3.2 非目标

第一阶段不应承担以下内容：

- 自研容器编排系统；
- 自研 GPU 驱动、虚拟化或硬件隔离；
- 完整替代 Kubernetes Scheduler；
- 生产级身份认证、计费、审计和权限管理；
- 多地域调度；
- 实际执行大规模分布式训练；
- 完整的 GPU 分时共享、MIG 管理或虚拟 GPU；
- 复杂的机器学习调度算法；
- Web 管理台优先开发。

这些内容可以成为后续研究方向，但不能掩盖项目的核心：调度政策、资源约束和可验证结果。

## 4. 目标工作负载

### 4.1 分布式训练任务

特点：需要多个 GPU 同时可用，通常运行时间较长，可能支持 checkpoint 和恢复，对 GPU 位置与网络拓扑较敏感。

关注点：整体调度、拓扑感知、排队时间、抢占恢复成本。

### 4.2 在线推理任务

特点：通常是长期驻留或对启动时间、尾延迟更敏感；可能需要较稳定资源，不宜频繁中断。

关注点：优先级、资源保留、快速调度、低延迟。

### 4.3 离线批处理任务

特点：可延迟、可暂停、可重试，对实时性要求较低。

关注点：填补资源碎片、提高整体利用率、作为可抢占候选。

### 4.4 交互式实验任务

特点：规模小、用户等待敏感、执行时间可能短，也可能因实验失败而频繁重试。

关注点：公平性、快速反馈、小任务避免长期饥饿。

## 5. 功能优先级

### P0：模拟器与资源/工作负载建模

目标：在没有真实 GPU 集群的前提下，先验证调度问题和策略差异。

应完成：

- 可配置的虚拟 GPU 集群；
- 节点、GPU 容量、基础拓扑和状态描述；
- 任务到达、运行、完成、失败、取消等生命周期；
- 训练、推理、批处理等任务类别；
- 用户、队列或租户标识；
- 可重复的 workload trace；
- 任务排队、放置、拒绝和完成结果；
- 基础资源利用率与等待时间统计。

P0 是整个项目的地基。没有模拟器，后续很容易把“偶然结果”误认为策略优势。

### P1：基础调度策略对照

目标：实现少量可解释的策略，并比较取舍。

优先策略：

- FIFO：最简单的基线；
- Best-effort / First-fit：快速放置的基线；
- Bin Packing：尽量集中分配以减少碎片；
- Fair Share：在用户或队列之间维护相对公平；
- Priority：让高优先级任务得到更快服务。

此阶段重点不是寻找“最优算法”，而是把不同目标之间的冲突量化出来。

### P2：AI 感知调度能力

目标：让调度器开始理解 AI 工作负载的特殊约束。

应完成：

- 多 GPU 任务的整体调度；
- 拓扑偏好表达与放置倾向；
- 训练、推理、批处理的差异化优先级；
- 资源配额；
- 任务可抢占性标记；
- 大任务与小任务的资源碎片问题分析。

P2 是 ClusterPilot 从普通任务队列升级为 AI 集群调度项目的关键。

### P3：抢占、重试与控制面可靠性

目标：模拟真实集群中的资源变化和失败。

应完成：

- 高优先级任务触发低优先级任务让出资源；
- 可恢复任务的暂停与恢复语义；
- 节点失效或 GPU 不可用时的任务处理；
- 重试策略和失败上限；
- 租约或心跳概念验证；
- 任务状态可追踪、可解释。

P3 的重点是状态一致性和可预测性，而不是复杂容错细节。

### P4：服务化、可观测性与策略评估

目标：把调度器从本地模拟程序变成可调用、可观测的控制面原型。

应完成：

- 提交、取消、查询任务的最小 API；
- 查询集群资源状态；
- 策略切换与实验配置；
- Prometheus 指标；
- 日志与关键决策记录；
- workload 回放；
- 实验报告和可视化。

### P5：Kubernetes 集成或适配层（可选）

目标：验证概念可以接近真实云原生环境，但不把 Kubernetes 集成变成项目主体。

可选方向：

- 将 ClusterPilot 的调度决策映射到 Kubernetes 任务对象；
- 读取或模拟节点 GPU 资源；
- 对接本地 kind、minikube 或小规模测试集群；
- 与 Volcano、Kubernetes Scheduler 的概念进行对照；
- 将 ClusterPilot 作为策略模拟器或外部调度决策服务。

若没有足够时间，P5 可以不做。P0 至 P4 已足以形成完整的系统项目。

## 6. 推荐技术栈

### 6.1 控制面语言：Go

**Go** 用于调度器、任务控制、API、状态管理、并发协调和可观测性集成。

选择原因：

- Kubernetes、Volcano、容器与云原生生态大量采用 Go；
- 适合长运行控制面、并发任务和网络服务；
- 工具链成熟，构建、测试和交付成本低；
- 与 C++/CUDA 的 BlockServe 形成能力互补。

不建议使用 C++ 编写 ClusterPilot 控制面。该项目的核心价值不在极限执行性能，而在资源策略、状态管理和云原生工程能力。

### 6.2 状态存储：PostgreSQL

**PostgreSQL** 用于持久化任务、租户、队列、资源快照和调度结果。

选择原因：

- 事务和查询能力足够支撑控制面状态；
- 便于做实验回放、状态审计和报表；
- 可避免早期引入分布式数据库带来的额外复杂度。

第一阶段可采用内存状态加可选落盘，P4 再要求 PostgreSQL 完整接入。

### 6.3 缓存与短期协调：Redis（后置）

**Redis** 仅在 P3/P4 后考虑，用于短期队列、租约、热点状态或实验加速。

不建议 P0/P1 就依赖 Redis。调度策略正确性应先在进程内和可重复模拟环境中验证。

### 6.4 事件机制：先轻后重

P0 至 P3 优先使用进程内事件、持久化状态和明确的控制循环。

只有在需要拆分服务、回放事件或展示异步控制面时，再引入 **NATS JetStream** 一类轻量事件系统。

不建议第一阶段使用 Kafka。Kafka 适合高吞吐、长期事件流场景，但对学生项目而言会显著增加运维复杂度，且无法直接证明调度策略能力。

### 6.5 可观测性

- **Prometheus**：指标采集；
- **Grafana**：实验结果和运行状态展示；
- **OpenTelemetry**：P4 以后用于请求链路与关键决策追踪；
- **结构化日志**：用于解释调度决策和状态转换。

### 6.6 部署与开发环境

- **Docker / Docker Compose**：本地环境一致性；
- **Makefile 或 Taskfile**：统一开发、测试、回放、压测命令；
- **GitHub Actions 或同类 CI**：单元测试、静态检查、模拟回归；
- **Python**：生成 workload、运行实验、分析与可视化；
- **Kubernetes / kind / minikube**：仅作为后期适配或演示环境，不应阻塞核心开发。

## 7. 实施方式

### 7.1 Simulator-first

第一原则是：先做高保真模拟器，再做服务化或 Kubernetes 对接。

原因：

- 你没有真实大规模 GPU 集群；
- 调度策略需要大量可控实验，而不是偶发的本地运行；
- 模拟器更容易注入节点失败、任务取消、突发流量和资源碎片；
- 同一 trace 可重复运行，便于公平比较不同策略。

模拟器不是“简化版项目”，而是 ClusterPilot 的核心实验平台。

### 7.2 Policy-first

优先实现少量、可解释、可对照的策略，不追求复杂算法。

每引入一项策略，都应明确：

- 它优化什么；
- 可能牺牲什么；
- 与什么基线比较；
- 在哪些 workload 下可能反而变差；
- 如何用指标验证。

例如，Bin Packing 可能提高资源利用率，但可能让部分任务等待更久；优先级调度可能改善重要任务的响应，却可能让低优先级任务饥饿。

### 7.3 先做“决策”，再做“执行”

ClusterPilot 的核心是决定谁获得什么资源、何时获得、资源不足时怎么办。

因此开发顺序应是：

1. 工作负载和资源状态；
2. 决策策略；
3. 状态转换；
4. 失败与抢占；
5. 观测与回放；
6. 外部 API；
7. Kubernetes 适配。

不要先做 UI、部署面板或 Kubernetes Operator。

### 7.4 与 BlockServe 的关系

两项目可保持独立仓库，但共享一组概念：

- BlockServe 暴露推理工作负载的资源轮廓，例如模型大小、显存预算、并发需求、延迟等级；
- ClusterPilot 把这类资源轮廓作为调度输入；
- BlockServe 的 benchmark 数据可以为 ClusterPilot 的推理 workload trace 提供更真实的参数；
- ClusterPilot 不直接管理 BlockServe 的内部缓存或 kernel。

这能体现你理解系统分层：单卡 Runtime 与集群控制面应协作，但不应混成一个巨型项目。

## 8. 评估维度

### 8.1 效率

- GPU 利用率；
- 空闲资源比例；
- 资源碎片程度；
- 已完成任务数；
- 任务吞吐；
- 大任务可成功调度比例。

### 8.2 延迟

- 任务平均排队时间；
- P50、P95、P99 排队时间；
- 高优先级任务启动时间；
- 交互任务响应时间。

### 8.3 公平性

- 不同用户/队列的资源获得比例；
- 长时间窗口内的资源偏差；
- 低优先级任务饥饿情况；
- 大任务与小任务之间的完成率差异。

### 8.4 稳定性与可恢复性

- 节点失败后受影响任务数；
- 重试成功率；
- 抢占后恢复率；
- 异常状态下是否有资源泄漏；
- 决策是否可解释和可追踪。

### 8.5 调度开销

- 每轮调度耗时；
- workload 增大后策略耗时变化；
- 状态持久化与查询开销；
- 观察与日志对控制面吞吐的影响。

## 9. 实验设计

至少准备以下实验集：

### 9.1 基线策略实验

在相同资源和 trace 下，对比 FIFO、First-fit、Bin Packing、Fair Share、Priority。

重点观察：吞吐、等待时间、公平性和碎片。

### 9.2 多 GPU 整体调度实验

设置不同规模的训练任务，比较“允许部分启动”与“整体启动”的资源占用和完成效率。

重点观察：被部分占用的资源是否造成长时间浪费，大任务成功率是否下降。

### 9.3 突发高优先级任务实验

在集群接近满载时注入推理或交互任务。

重点观察：抢占策略是否改善高优先级任务响应，同时评估对低优先级任务的代价。

### 9.4 拓扑偏好实验

设置不同节点和 GPU 域，比较“任意可用资源”与“优先局部放置”策略。

重点观察：资源利用率、可放置率和预设通信成本模型下的任务效率。

### 9.5 故障注入实验

随机让节点或 GPU 不可用，观察任务状态、重试、资源回收和最终完成情况。

实验不必真的让物理 GPU 宕机；模拟器中注入故障即可。

## 10. 测试与质量要求

### 10.1 单元测试

重点覆盖：

- 资源是否重复分配；
- 任务完成、取消、失败后资源是否归还；
- 配额是否被正确限制；
- 抢占是否遵守优先级和可抢占性；
- 多 GPU 任务是否满足整体调度要求；
- 失败重试是否符合次数和条件限制；
- 策略切换是否不破坏状态一致性。

### 10.2 属性测试与随机测试

生成随机任务序列、随机资源状态和随机故障，验证不变量：

- 分配资源不超过集群容量；
- 同一 GPU 不会同时分配给互斥任务；
- 已释放资源可再次使用；
- 任务状态不会出现非法跳转；
- 配额不会因重试或抢占被绕过。

### 10.3 回归测试

保留固定 workload trace。每次改动策略或状态逻辑后，检查关键指标和行为是否发生不可解释变化。

## 11. 主要风险与控制方式

### 风险：项目变成 Kubernetes 二次实现

控制方式：明确 ClusterPilot 聚焦“策略模拟、AI 工作负载语义和控制面行为”，不重做容器生命周期、网络、存储和完整调度框架。

### 风险：早期引入太多中间件

控制方式：P0/P1 用内存模拟；持久化、Redis、事件总线、Kubernetes 都后置。

### 风险：没有真实 GPU 集群导致项目不可信

控制方式：使用可重复 trace、清楚的资源模型、故障注入和对照实验。说明哪些结论来自模拟，哪些来自小规模验证。

### 风险：策略复杂但无法解释

控制方式：只实现少量策略，每一项都有目标、代价、基线和指标。

### 风险：只展示调度结果，不展示状态可靠性

控制方式：把取消、失败、重试、抢占、资源回收纳入验收标准。

## 12. 最终交付物

最终仓库应至少包含：

- 调度核心；
- 集群与工作负载模拟器；
- 多种调度策略；
- 任务、资源、租户和队列的状态模型；
- 失败与抢占的模拟能力；
- workload trace；
- benchmark 和可视化脚本；
- 最小 API 或 CLI；
- Prometheus 指标；
- 实验报告；
- 策略取舍说明；
- Kubernetes/Volcano 对照说明；
- 已知限制与后续工作。

最终演示不应是“一个网页显示 GPU 卡片”，而应是：

> 给定相同集群和相同工作负载，ClusterPilot 如何在不同策略下改变资源利用率、等待时间、公平性、任务完成率和高优先级响应。

## 13. 建议参考资料

优先参考以下资料的概念、约束和评估方法：

- Kubernetes Scheduler 官方文档；
- Kubernetes GPU Scheduling 官方文档；
- Volcano 的 Gang Scheduling、Queue、Capacity、Binpack、异构设备调度文档；
- NVIDIA GPU Operator 关于 GPU sharing 的文档；
- NCCL collective communication 文档；
- Kubernetes scheduler framework 与调度器代码中的队列、filter、score、bind 等概念；
- 公开集群调度 trace 和相关调度系统论文。

阅读时应重点理解：普通容器调度与 AI 作业调度的差异、Gang Scheduling 的必要性、队列公平性和资源碎片的取舍，而不是复制某个系统的内部实现。
