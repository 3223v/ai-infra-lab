# BlockServe 项目说明书

## 1. 项目定位

**项目名称**：BlockServe  
**项目类型**：单 GPU、大语言模型推理运行时（LLM Inference Runtime）  
**目标岗位**：GPU Runtime、AI 推理引擎、异构计算、AI Infra、国产 GPU 软件栈、性能工程

BlockServe 不是通用聊天应用，也不是对现有推理框架的简单包装。它的目标是在有限显存条件下，完成一个可运行、可测量、可解释的 LLM 推理服务原型，重点展示以下能力：

- 对自回归推理过程的理解；
- 对 GPU 显存、KV Cache 与请求生命周期的管理能力；
- 对并发请求调度与吞吐、延迟取舍的理解；
- C++、CUDA、性能分析与工程化能力；
- 用实验而不是口头描述验证系统优化效果的能力。

项目应以“运行时能力”为中心。模型支持范围、界面、协议兼容性均为次要目标。

---

## 2. 要解决的问题

大语言模型服务端推理有两个现实约束：

1. 单个请求的生成过程持续占用 KV Cache，且上下文越长、并发越高，显存压力越大；
2. 请求长度、到达时间和生成长度不一致，静态批处理会造成等待、显存浪费或 GPU 空转。

BlockServe 聚焦在受限显存场景下回答以下问题：

- 如何让多个长度不同的请求共存，而不过度预留显存？
- 如何让请求动态加入、完成和退出，而不等待整个批次结束？
- 如何在首 token 延迟、每 token 延迟、总吞吐和显存利用率之间做取舍？
- 如何在显存不足时做到可预测地拒绝、排队、取消或回收，而不是随机 OOM？
- 如何通过基线对照证明优化并非“主观感觉更快”？

项目不以追平 vLLM、TensorRT-LLM、llama.cpp 的通用性或性能为目标。它的价值在于清楚展示核心问题、实现路径和实验结果。

---

## 3. 目标与非目标

### 3.1 核心目标

- 在本地 4GB 显存 GPU 上运行一个小型 decoder-only 模型；
- 支持多请求推理与流式输出；
- 支持按需增长的 KV Cache 管理方式；
- 支持连续批处理，而非“一个批次跑完再处理下一批”；
- 提供稳定的请求状态管理、超时、取消和资源回收；
- 对推理延迟、吞吐、显存与调度行为进行可观测和可复现实验；
- 形成可以被招聘者、导师或开源社区理解的技术文档与实验报告。

### 3.2 非目标

下列内容明确不作为第一阶段的目标：

- 多机多卡或跨节点通信；
- 大规模模型训练；
- 多模型路由；
- 多模态输入；
- Agent、RAG、工作流编排；
- 完整 OpenAI API 兼容；
- 面向所有 Hugging Face 模型的自动兼容；
- 与工业推理框架进行绝对性能竞争；
- 复杂量化、MoE、投机解码、前缀缓存等高级功能。

这些能力可以在项目成熟后作为扩展项，但不应影响核心链路交付。

## 4. 使用场景与工作负载

项目应围绕四类典型工作负载设计和验证：

### 4.1 短对话请求

特点是 prompt 短、输出短、用户更关注首 token 延迟。该场景用于验证请求进入系统后的排队与启动效率。

### 4.2 混合长度请求

同一时间存在短 prompt、长 prompt、短输出和长输出请求。该场景用于验证连续批处理能否减少静态批处理带来的等待和填充浪费。

### 4.3 长 prompt 压力请求

请求输入较长、输出较短。该场景用于观察 prefill 阶段是否垄断 GPU，以及是否需要将长输入拆分为可调度的片段。

### 4.4 显存紧张请求

人为限制可用于 KV Cache 的显存预算，观察不同缓存策略下的请求承载量、拒绝率、回收行为和尾延迟。

这些 workload 不需要真实业务数据。重点是请求到达模式、输入长度、输出长度和资源压力具有可控性与可重复性。

## 5. 功能优先级

项目应严格按优先级推进。未完成高优先级内容前，不进入低优先级功能。

### P0：运行时问题建模与模拟验证

目标：先脱离模型和 GPU，验证资源管理与调度规则是否合理。

应完成：

- 请求生命周期定义；
- 请求到达、排队、执行、完成、取消、超时等行为模拟；
- 固定显存预算下的 KV Cache 占用估算；
- 静态预留缓存与按需增长缓存的对照；
- 固定批处理与连续批处理的对照；
- 负载 trace 回放；
- 基础指标统计。

P0 的产物是一个模拟器和一份实验说明。它应能证明：即使不运行真实模型，调度与缓存策略已经具有可测量的差异。

### P1：单请求真实推理与正确性验证

目标：让小模型在 C++/CUDA 路径上正确完成生成。

应完成：

- 加载一个固定架构的小型 decoder-only 模型；
- 完成单请求推理和 token 生成；
- 支持基础 KV Cache；
- 与 Python/PyTorch/Transformers 参考实现对比中间结果或 logits；
- 建立可自动运行的正确性测试。

P1 的验收不以性能为主，而是以正确性、可重复性和错误定位能力为主。

### P2：显存受控的 KV Cache 管理

目标：让 KV Cache 从“连续、一次性、粗粒度占用”转向“按块、按需、可回收”的管理。

应完成：

- 固定预算下的缓存容量管理；
- 请求增长时的按需扩容；
- 请求结束、取消、超时后的及时回收；
- 内存不足时的可预测处理，而非随机崩溃；
- 连续缓存与分页式缓存的正确性与资源利用率对照；
- 长时间运行下的内存泄漏检查。

P2 是项目最重要的记忆点之一。完成后，项目才具备“推理 Runtime”而非“模型 Demo”的特征。

### P3：Continuous Batching 与调度

目标：让多个请求在不同阶段动态共享 GPU，而不是静态地组成一个固定批次。

应完成：

- 请求队列与准入控制；
- Prefill 和 Decode 的不同处理策略；
- Decode 优先的基本调度规则；
- token 预算或资源预算约束；
- 请求完成后立即释放资源并接纳新请求；
- 混合 workload 下的吞吐、延迟和显存对照实验。

P3 完成后，BlockServe 的主体能力基本完整。

### P4：可观测性、压测与性能分析

目标：把“系统做了什么”转化为可解释的数据。

应完成：

- 请求级指标：排队时间、首 token 延迟、生成速度、完成原因；
- 调度级指标：队列长度、每轮调度量、拒绝量、取消量；
- 缓存级指标：已用容量、峰值占用、分配失败、回收次数；
- GPU 级指标：显存使用、执行耗时、关键阶段耗时；
- 可复现 benchmark；
- Profiling 记录和结论；
- 结果分析报告。

P4 决定项目是否能进入简历。没有数据、对照和结论的性能项目，可信度有限。

### P5：可选扩展

仅在 P0 至 P4 稳定后再考虑：

- Chunked Prefill；
- 简单前缀复用；
- 轻量 HTTP/SSE 服务；
- 轻量量化；
- 多种模型权重格式；
- CUDA Graph；
- 不同后端的兼容性探索。

扩展功能应服务于已有实验问题，而不是为了堆功能列表。

## 6. 推荐技术栈

### 6.1 核心语言

**C++20**：用于运行时主体、显存资源管理、调度、模型执行控制、服务生命周期和性能关键路径。

选择原因：

- 与 GPU Runtime、推理引擎和系统软件岗位高度相关；
- 对内存、并发和资源生命周期控制更直接；
- 便于与 CUDA、cuBLASLt 等生态集成；
- 适合把项目做成可独立运行的工程，而不是 Python 原型。

**Python**：用于模型导出、tokenizer、参考结果、trace 生成、压测、数据分析和可视化。

选择原因：

- 模型生态和实验工具最成熟；
- 不应承担运行时核心逻辑；
- 可作为正确性基准，减少底层开发阶段的定位成本。

### 6.2 GPU 与数值计算

**CUDA Runtime API**：负责设备内存、stream、事件、错误检查和 GPU 执行控制。

**cuBLASLt**：用于通用矩阵乘法和线性层计算。第一阶段不应自行实现矩阵乘法。

**自定义 CUDA kernel**：优先用于有明确学习价值且可控的算子，例如归一化、位置编码、采样或分页缓存访问路径。是否继续深入优化，应由 profiling 数据决定。

不建议第一阶段引入完整 Tensor Library、通用图编译器或大量第三方推理引擎。它们会掩盖你真正需要理解的运行时问题。

### 6.3 构建、测试与工程化

- **CMake + Ninja**：跨平台构建与 CUDA 工程管理；
- **GoogleTest 或 Catch2**：单元测试与回归测试；
- **fmt + spdlog**：结构化日志和错误定位；
- **clang-format / clang-tidy**：代码风格和静态检查；
- **GitHub Actions 或同类 CI**：至少覆盖 CPU 侧单元测试、格式检查和模拟器测试。

### 6.4 模型与实验工具

- **PyTorch**：参考实现和数值正确性验证；
- **Transformers**：模型加载、tokenizer 和参考生成；
- **Safetensors**：权重读取或导出辅助；
- **pytest**：Python 工具链测试；
- **Matplotlib**：绘制实验结果。

### 6.5 性能分析工具

- **Nsight Systems**：观察 CPU 调度、CUDA 调用、kernel 执行和同步关系；
- **Nsight Compute**：分析重点 kernel；
- **NVTX**：标记请求处理、调度、缓存分配、prefill、decode 等阶段；
- **nvidia-smi / CUDA event**：基础资源和耗时观测。

## 7. 在 4GB 显存下的实施原则

4GB 显存不适合追求大模型规模，但非常适合研究资源约束下的调度和缓存问题。

实施原则：

- 选择小型模型，优先保证可重复运行；
- 控制最大上下文长度与并发数量；
- 显式预留安全余量，不以“榨干显存”为目标；
- 通过人为设定较紧的缓存预算制造压力场景；
- 重点比较策略差异，而不是追求绝对 tokens/s；
- 将调度模拟器作为真实 GPU 实验的补充，而不是替代。

项目报告必须诚实区分：哪些结论来自真实 GPU 测试，哪些结论来自模拟实验。两者都有效，但不能混淆。

---

## 8. 实施方式

### 8.1 先建立可验证基线

每个优化都必须有对照对象。至少保留：

- 单请求顺序执行；
- 固定批处理；
- 连续批处理但采用简单缓存策略；
- 连续批处理加分页式缓存策略。

没有基线，就无法判断一个功能是否真正带来收益。

### 8.2 每阶段只验证一个主要问题

例如：

- P0 验证调度和资源策略；
- P1 验证数值正确性；
- P2 验证缓存管理；
- P3 验证动态合批；
- P4 验证性能结论。

不要在同一阶段同时引入新模型、新算子、新调度算法和新服务协议。否则问题无法定位。

### 8.3 使用“先正确、后可测、再优化”的节奏

具体顺序应为：

1. 建立可运行的最小路径；
2. 建立与参考实现的一致性检查；
3. 建立 trace 和 benchmark；
4. 定位真正瓶颈；
5. 再进行针对性优化；
6. 用同一 workload 重跑实验。

### 8.4 文档与实验同步推进

每完成一个里程碑，应更新：

- 当前功能范围；
- 已知限制；
- 测试方法；
- 实验环境；
- 关键结论；
- 下一阶段不做什么。

这比最后补一篇 README 更有价值。

## 9. 指标与验收

### 9.1 正确性指标

- 输出 token 或 logits 与参考实现的差异；
- 缓存策略改变前后生成结果的一致性；
- 请求取消、超时、完成后的资源是否正确回收；
- 长时间运行后显存是否出现异常增长；
- 极端 workload 下系统是否可预测地拒绝或排队。

### 9.2 性能指标

- TTFT：首 token 延迟；
- TPOT：后续 token 间隔；
- 总输出 tokens/s；
- 请求吞吐；
- P50、P95、P99 延迟；
- 峰值显存使用；
- 缓存利用率与拒绝率；
- 排队时间；
- 完成请求数。

### 9.3 最低验收标准

项目可被视为完成，至少应满足：

- 能在目标机器上稳定运行小模型；
- 能处理多个并发请求；
- 能明确展示至少一种缓存策略和一种调度策略；
- 有正确性测试；
- 有四类以上 workload；
- 有基线对照；
- 有可复现的实验脚本与环境说明；
- 有性能分析结论；
- 有清楚的已知限制。

## 10. 主要风险与控制方式

### 风险：过早陷入 CUDA kernel 优化

控制方式：先使用成熟矩阵库完成主链路，只优化被 profiling 证明为瓶颈的部分。

### 风险：模型兼容和权重格式消耗过多时间

控制方式：第一阶段只支持一个固定模型族和受控导出格式，不做通用加载器。

### 风险：4GB 显存导致实验规模有限

控制方式：强调受限资源下的策略比较；使用模拟器扩展 workload；在项目成熟后再短期使用云 GPU 做补充验证。

### 风险：功能很多但没有结论

控制方式：每项功能都必须对应一个明确问题、一组基线和一个可量化结论。

### 风险：项目成为现有框架的低配复刻

控制方式：明确自己的研究边界：受限显存、请求生命周期、KV Cache 管理和调度行为，而不是通用推理产品功能。

最终仓库应至少包含：

- 运行时核心代码；
- CPU 侧调度/缓存模拟器；
- Python 模型导出与参考工具；
- 正确性测试；
- benchmark trace 和配置；
- 性能分析脚本与图表；
- 项目架构概览；
- 实验报告；
- 已知限制与后续计划；
- 可复现环境说明。

最终演示应聚焦一件事：在相同模型、相同硬件、相同请求 trace 下，BlockServe 如何通过缓存和调度策略改善资源利用或服务行为。
